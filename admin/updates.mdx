Collabase provides a dedicated update script — `deployment/update.sh` — that handles the full update process safely. Never use `install.sh` to update an existing installation.

## Running the update

```bash
bash deployment/update.sh
```

The script performs the following steps automatically:

1. **Migrates old variable names** — if your `.env` still uses `HELVI_*` variable names from an earlier version, they are renamed to `COLLABASE_*` automatically.
2. **Provisions any missing secrets** — new releases may require additional environment variables. Any missing keys (e.g. `MAIL_ENCRYPTION_KEY`, `AI_ENGINE_URL`) are generated and appended to `.env` so you never hit a startup error after an upgrade.
3. **Creates a pre-update database backup** — a compressed dump of the current database is saved to `deployment/backups/collabase-<timestamp>.sql.gz` before any images are pulled.
4. **Pulls the latest images** from the GitHub Container Registry (`ghcr.io/collabase/`).
5. **Restarts all containers** with `--remove-orphans` to remove containers for services that no longer exist in the current release.
6. **Waits for the app to be healthy** — polls `/api/health` for up to 5 minutes, printing progress dots while waiting.

<Note>
  Database migrations are applied automatically on container startup — the same `prisma migrate deploy` that runs on first boot also runs on every update. You do not need to apply migrations manually.
</Note>

## Pre-update backup

`update.sh` always creates a backup before pulling new images:

```
deployment/backups/collabase-20260101-120000.sql.gz
```

To restore from this backup if an update goes wrong:

```bash
# 1. Stop the app (keep the database running)
docker compose -f deployment/docker-compose.release.yml stop collabase-app

# 2. Restore the compressed dump
gunzip -c deployment/backups/collabase-20260101-120000.sql.gz | \
  docker exec -i collabase-postgres psql -U collabase collabase

# 3. Restart with the previous image (pin the image tag first)
docker compose -f deployment/docker-compose.release.yml up -d
```

<Warning>
  If the backup step fails (e.g. the database container is not running), `update.sh` prints a warning and continues. Verify the database is healthy before updating in that case.
</Warning>

## Security warnings during update

The script checks for common security issues and warns you inline:

| Warning | What to do |
|---|---|
| `POSTGRES_PASSWORD` is set to `collabase` | Change the password in both `.env` and PostgreSQL — see [Backup & Restore](/admin/backup) for the command |

## After the update

Once the update completes, the script prints a summary:

```
✅ Collabase updated successfully!

  URL:     https://collabase.example.com
  Version: sha256:abc123 → sha256:def456
  Backup:  deployment/backups/collabase-20260101-120000.sql.gz
```

Navigate to your Collabase URL and verify everything is working. If users report issues, check the application logs:

```bash
docker compose -f deployment/docker-compose.release.yml logs -f collabase-app
```
